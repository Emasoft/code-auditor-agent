# l10n lens

## key
l10n

## fire-when
Code performing locale-aware formatting (dates, numbers, currencies, plurals). Greppable markers: `Intl.DateTimeFormat`, `Intl.NumberFormat`, `Intl.PluralRules`, `toLocaleString`, `toLocaleDateString`, `Luxon`/`DateTime.toLocaleString`, `moment`, `dayjs`, `date-fns`, `babel.dates.format_date`, `format_currency`, `format_decimal`, `Accept-Language`, `Intl.Locale`, `{count, plural`, `{n, select`, ICU MessageFormat, `new Date(`, `datetime.now(`, hardcoded format strings (`"MM/DD/YYYY"`, `"DD/MM/YYYY"`), hardcoded currency symbols (`"$"`, `"€"`) or thousand separators. Locale/i18n config files, currency/date display in views/templates. (Translation-string presence/extraction is the separate i18n lens, not this one.)

## checklist
Audit ONE file for locale-aware formatting correctness. Layer is `structural`. For each finding emit Severity (MUST-FIX | SHOULD-FIX | NIT), Confidence (HIGH | MEDIUM | LOW), and Category (one of: date-format | number-format | currency | timezone | message-format | calendar | locale-negotiation | rtl | length-assumption).

- Date formatting: new date displays must use a locale-aware formatter (`Intl.DateTimeFormat`, `Luxon.DateTime.toLocaleString`, `babel.dates.format_date`) — NOT hardcoded format strings like `"MM/DD/YYYY"`. [Category: date-format, SHOULD-FIX]
- Number formatting: numbers use `Intl.NumberFormat` / locale-aware formatters; hardcoded `","` thousand-separator or manual grouping. [Category: number-format, SHOULD-FIX]
- Currency formatting: currencies use locale-aware formatters; hardcoded `"$"` (or any fixed currency symbol / placement) is wrong. [Category: currency, MUST-FIX]
- Timezone handling: date arithmetic must use UTC internally and convert to the user's timezone via the framework on display; `new Date()` / `datetime.now()` used as "now in user TZ" WITHOUT explicit conversion. [Category: timezone, MUST-FIX]
- ICU MessageFormat: translations with placeholders use ICU syntax (`{count, plural, ...}`) when the framework supports it, else the framework's plural/select API; a literal string-concat workaround for pluralization is wrong. [Category: message-format, SHOULD-FIX]
- Calendar systems: locales using non-Gregorian calendars (Thai Buddhist, Japanese imperial, Persian, Hijri) should render dates in their own calendar via `Intl.DateTimeFormat(locale, { calendar })`; missing support. [Category: calendar, NIT]
- Locale negotiation: server picks locale from `Accept-Language`, intersected with the app's supported locales, with a default fallback; a hardcoded `en-US` (or single hardcoded locale) is wrong. [Category: locale-negotiation, SHOULD-FIX]
- RTL-aware formatting: bidi-isolation chars (FSI/PDI, `⁨...⁩`) wrap user-supplied content embedded inside framed/templated text; missing isolation. [Category: rtl, NIT]
- String-length assumptions: UI layouts that assume English-length strings (fixed-width button labels, truncating ellipsis on translatable text). [Category: length-assumption, SHOULD-FIX]
- Combined high-severity rule: hardcoded currency symbols AND user-TZ derived from `new Date()` without conversion are each independently MUST-FIX.
