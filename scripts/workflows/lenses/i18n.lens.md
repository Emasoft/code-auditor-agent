# i18n lens

## key
i18n

## fire-when
Project ships translation/locale files (`locales/**`, `i18n/**`, `*.po`, `*.pot`, `messages.json`, `en.json`/`*.json` under a locale dir, `*.arb`, `*.ftl`) OR source imports an i18n lib (`i18next`, `react-i18next`, `next-intl`, `vue-i18n`, `formatjs`/`react-intl`, `gettext`, `Qt tr`, `gnu gettext`). Also fires on source files containing i18n call markers: `t(`, `<Trans`, `i18n.t(`, `tr(`, `useTranslation`, `gettext`, `_(`. Scope is i18n only — locale-specific date/currency/number formatting belongs to the l10n lens.

## checklist
Apply to ONE file. Severities and categories are exact; emit them verbatim. Layer is always `structural`.

- **[hardcoded-string] MUST-FIX:** New user-visible strings in source (button labels, error messages, dialog/UI text) that bypass the i18n layer (`t(...)`, `<Trans>`, `i18n.t(...)`, `tr(...)`). Any raw user-facing literal not wrapped by a translation call.
- **[concatenation] MUST-FIX:** Translations built via string concatenation (`t("hello") + " " + name`) — loses word order in some locales. Must use interpolation (`t("hello_name", { name })`) instead.
- **[untranslated] MUST-FIX:** A locale file ships an empty or placeholder value in production — `"key": ""` or `"key": "TODO"`.
- **[plurals] MUST-FIX:** Strings with counts must use the framework's plural API (`{count, plural, one {...} other {...}}`, `t("key", { count })`). A manual `count === 1 ? "x" : "xs"` ternary for pluralization is a violation.
- **[gender] SHOULD-FIX:** When the locale supports grammatical gender / grammatical-number, the framework's selectors must be used. Missing gender/number selectors.
- **[rtl] SHOULD-FIX:** New layouts must use CSS logical properties (`margin-inline-start`, not `margin-left`) OR pass through a RTL-aware utility. Physical directional properties (`margin-left:`, `padding-right:`) in source-of-truth styles are a violation.
- **[key-consistency] SHOULD-FIX:** A new translation key added to one locale (e.g. `en.json`) but missing from the other locale files. (CI often catches this — surface only if the project lacks the CI check.)
- **[naming] NIT:** Translation keys must follow the codebase's existing convention (dotted / underscored / namespaced).

Confidence per finding: HIGH | MEDIUM | LOW.
